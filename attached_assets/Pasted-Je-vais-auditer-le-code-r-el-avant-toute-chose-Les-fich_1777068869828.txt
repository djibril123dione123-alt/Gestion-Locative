## 1. Audit rapide du code réel

**`Paiements.tsx` — split('T')**

Document index 121, ligne dans `handleSubmit` :
```typescript
const moisConcerne = new Date(formData.mois_display + '-01')
    .toISOString()
    .split('T'); // ← pas de [0]
```
Et plus bas dans `data` :
```typescript
mois_concerne: moisConcerne, // tableau string[], pas string
```
**BUG CONFIRMÉ.** `moisConcerne` est `string[]`, pas `string`. Ça insère un tableau en DB → la contrainte `date NOT NULL` ou le type `date` dans PostgreSQL va soit rejeter, soit stocker `["2024-01-01"]`.

**`Paiements.tsx` — confirm() natif**

```typescript
if (!confirm('Êtes-vous sûr de vouloir supprimer ce paiement ?')) return;
```
**CONFIRMÉ**, ligne dans `handleDelete`.

**`Equipe.tsx` — ToastContainer props**

Doc index 7, fin du fichier :
```typescript
<ToastContainer toasts={toast.toasts} onClose={toast.removeToast} />
```
Doc index 2, `ToastContainer` signature :
```typescript
interface ToastContainerProps {
  toasts: Array<...>;
  onRemove: (id: string) => void;
}
```
La prop s'appelle `onRemove`, pas `onClose`. **BUG CONFIRMÉ** — TypeScript devrait normalement le détecter si `strict` est activé... mais le fichier dit que ça compile. Vérifions `useToast`.

Doc index 122, `useToast` retourne :
```typescript
return { toasts, addToast, showToast: addToast, removeToast, success, error, warning };
```
Pas de `onClose`. Donc `toast.removeToast` existe, mais la prop de `ToastContainer` s'appelle `onRemove`. Si TypeScript compile quand même, c'est que soit le composant a une prop alternative non documentée, soit il y a un type `any` quelque part.

**`LoyersImpayes.tsx` — ToastContainer**

Doc index 20 :
```typescript
<ToastContainer toasts={toast.toasts} onClose={toast.removeToast} />
```
**MÊME BUG**, `onClose` au lieu de `onRemove`.

**`usePlanLimits` — agences sans subscription**

Doc index 61, `usePlanLimits.ts` actuel :

**Ensuite** dans le même commit : corriger `onClose → onRemove` dans `Equipe.tsx` et `LoyersImpayes.tsx` car c'est une ligne chacun et ça ne risque rien.

**Le confirm() de Paiements** sera un mini-chantier suivant car il demande d'ajouter un state + ConfirmModal.
