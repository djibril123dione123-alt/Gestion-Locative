# Audit total Samay Keur — Rapport complet

Je vais auditer systématiquement tous les fichiers fournis dans le contexte de la conversation.

---

## A. Vision globale du projet

**Niveau de maturité : MVP fonctionnel avancé — pré-prod fragile**

**Forces réelles :**
- Architecture multi-tenant cohérente avec `agency_id` sur toutes les tables métier
- RLS bien pensé structurellement, migrations idempotentes
- Refactorisation des types partiellement faite (`src/types/`, `src/lib/formatters.ts`)
- Console super_admin existante et fonctionnelle
- 16+ pages métier présentes, onboarding complet
- PDF génération fonctionnelle avec templates texte

**Faiblesses structurelles :**
- Logique métier mélangée dans les pages (pas de couche service)
- Types `any` nombreux dans les pages critiques (Paiements, Depenses, Commissions, TableauDeBordFinancierGlobal)
- Bugs de données silencieux confirmés (split T, onClose vs onRemove)
- Zéro test automatisé
- Zéro monitoring / observabilité
- Zéro CI/CD
- Pages métier nouvelles (Inventaires, Interventions, Calendrier, Documents) non battle-tested

**Verdict honnête : pré-prod fragile.** Vendable en démo, pas vendable en production sérieuse avec des clients payants.

## C. Ce qui est incomplet

**Partiellement branché mais non fiabilisé :**
- `feature_flags` : table créée, données insérées, mais **aucune page ne lit ces flags** pour conditionner l'affichage de modules
- `saas_config` : éditable en Console, mais `maintenance_mode` n'est jamais lu par l'app pour afficher une bannière
- `revenus` : table existante, DELETE dans Paiements, mais **aucun INSERT automatique** à la création d'un paiement (ni trigger, ni code)

- `owner_actions_log` : table et policy créées, appelée dans `Console.tsx` `logOwnerAction`, mais **si la policy exige super_admin et que les logs sont silencieux en cas d'erreur**, l'audit est vide

**Présent en DB mais non exploité :**
- `audit_logs` (trigger SQL) : les triggers existent sur toutes les tables critiques, mais la Console ne propose aucune vue de ces logs par agence

- `subscription_plans.features` jsonb : présent mais jamais lu pour conditionner des features
- `agencies.tags` : créé dans migration 20260425000002, utilisé dans Console pour badger VIP/À risque, mais non persisté (l'action "marquer VIP" n'existe pas dans le code Console fourni)

**Pensé mais non industrialisé :**
- Système de relances automatiques (loyers impayés) : détection présente, action manuelle seulement
- Export comptable : export Excel de `FiltresAvances` seulement, pas de vrai export comptable

## D. Risques cachés

1. **Migration conflicts sur agencies INSERT** : plusieurs migrations créent/suppriment des policies INSERT sur `agencies`. La dernière appliquée gagne — mais l'ordre d'application en prod peut diverger de l'ordre chronologique si les timestamps sont identiques ou mal ordonnés.

2. **Trigger double-insert agency_settings** : `create_agency_settings_on_agency_insert` crée automatiquement un row dans `agency_settings`. `Welcome.tsx` en crée aussi un. En prod, le premier à s'exécuter gagne, le second échoue. Si `Welcome.tsx` s'exécute avant le trigger (race condition async), la ligne est créée avec les bonnes données mais le trigger échoue silencieusement — ou l'inverse. Données de settings potentiellement vides.

3. **`check_plan_limits` bloque les agences legacy** : toute agence sans row dans `subscriptions` se voit appliquer les limites Basic (1 user, 3 immeubles, 10 unités). Si des données de test existent sans subscription associée, les admins ne peuvent plus créer de contenu.

4. **`AcceptInvitation` token perdu** : si l'utilisateur clique sur un lien d'invitation sans être connecté, il est redirigé vers Auth. Une fois connecté, l'app charge sur `/` sans token → invitation jamais acceptée.

5. **Realtime notifications fuite** : si `NotificationBell` est monté/démonté rapidement (ex: lors du chargement initial), plusieurs channels peuvent coexister momentanément avant le cleanup. Avec Supabase Realtime, les channels orphelins consomment des connexions.

6. **`Console.tsx` N+1 sur les KPIs** : `Promise.all([agRes, userRes, subRes, logRes])` charge jusqu'à 200 users + toutes les agences + subscriptions en une fois. À l'échelle (50+ agences), ce sera lent.

7. **PDF settings non mis en cache** : chaque clic "Générer PDF" déclenche 2 requêtes Supabase. Avec 10 PDF générés en séquence, c'est 20 requêtes inutiles.

8. **`slice(0, 36)` pour extraire contrat_id dans LoyersImpayes** : un UUID v4 fait 36 caractères, mais ce découpage est fragile. Si l'ID change de format (ex: CUID, nanoid), le bug est silencieux.

9. **`Depenses.tsx` aucun guard agency_id sur le DELETE** : `handleDelete` vérifie `if (!profile?.agency_id) return` mais le delete lui-même ne filtre pas par `agency_id` — uniquement par `id`. La RLS protège en théorie, mais si la RLS est trop permissive temporairement, cross-tenant delete possible.

10. **Bucket Storage non créé** : le bucket `documents` n'est pas créé programmatiquement. En prod fresh, le premier upload échoue sans message d'erreur explicite.

## H. Verdict final sans langue de bois

**Non, ce projet n'est pas finalisé.**

Il est dans un état qu'on peut appeler **"MVP démontrable mais non déployable sérieusement"**.

**Ce qui bloque concrètement :**

1. **Un bug de corruption de données actif** (split T) — chaque paiement créé ou modifié depuis la mise en place de cette page est potentiellement corrompu en DB.

2. **L'onboarding peut crasher** sur de nouveaux comptes à cause du double-insert `agency_settings` (Welcome + trigger auto).

3. **La fonctionnalité Documents est entièrement morte** en prod (bucket non créé).

4. **Les invitations d'équipe sont cassées** pour les utilisateurs non connectés (token perdu après redirect).

5. **Zéro visibilité en production** : pas de Sentry, pas de logs structurés, pas de monitoring. Un bug en prod est invisible.

6. **Zéro filet de sécurité** : pas de tests, pas de CI/CD, chaque modification peut casser silencieusement n'importe quoi.

**Ce qu'il faut pour dire "on peut lancer proprement" :**
- Phase 1 (stabilisation) complétée : ~2 jours
- Phase 2 (complétude fonctionnelle) complétée : ~5 jours
- Sentry + `.env.example` + README : ~1 jour
- Un test E2E manuel documenté des 5 flux critiques : ~1 jour

**Total réaliste : 10-15 jours de travail ciblé avant un lancement client sérieux.**

Le code est de bonne qualité globale pour un MVP rapide. L'architecture multi-tenant est saine. Mais la distance entre "ça marche en démo" et "c'est fiable pour de vrais clients" est encore significative.
